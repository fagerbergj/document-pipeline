Rules:
- Fix OCR errors: misread characters, broken ligatures, garbled words.
- Expand abbreviations, shorthand, and terse notes into full sentences where you understand the intent. Add a Markdown comment immediately after the expanded text noting what was expanded, e.g. `<!-- expanded: "w/" → "with" -->`.
- Where context makes the meaning clear, fill in implied information. Add a Markdown comment noting what was inferred, e.g. `<!-- inferred: assumed "mtg" refers to the Tuesday standup from context -->`. Do NOT use inline [added: …] markers.
- Improve readability: fix punctuation, capitalisation, and structure where it helps.
- Output as Markdown. Actively use Markdown features where they improve clarity: headings (`#`, `##`) for sections or topics, bullet or numbered lists for enumerations or steps, tables for comparisons or structured data, bold for key terms or action items. Use the format that best matches the content — don't force structure where none exists, but don't leave structure implicit when Markdown can make it explicit.
- Do NOT rephrase or reorder content. Preserve the author's structure.
- Do NOT invent specific facts (names, numbers, dates) not clearly implied.
- Ask clarifying questions for: illegible characters, ambiguous abbreviations, or phrases with meaningfully different interpretations. Be proactive.

Set confidence based on how certain you are of the output:
- "high": very few or no ambiguities, meaning is clear
- "medium": some uncertainty resolved via context, minor questions remain
- "low": significant ambiguities, illegible segments, or missing context

Respond with valid JSON only:

  "clarification_requests": [
    {"segment": "<exact text from OCR>", "question": "<specific question>"}
  ],

}
